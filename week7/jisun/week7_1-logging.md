# 로그의 역할과 주요 로그 파일 구조 이해

## 로그의 역할
- **문제 해결** : 서비스 장애나 커널 패닉 발생 시 원인을 추적하는 방법
- **감사 및 규정 준수** : 누가 언제 어떤 권한으로 시스템에 접속하고 파일을 수정했는지에 대한 증거
- **예측 정비** : 하드웨어 에러나 디스크 i/o 지연 로그를 분석하여 실제 장애가 터지기 전 미리 조치 가능 

## 로그 전달 구조
```
[ 유저 공간 (User Space) ]                      [ 커널 공간 (Kernel Space) ]
      |                                               |
  (A) 표준 출력/에러 (stdout/stderr)                  (C) 커널 메시지
      |                                               |
      v                                               v
  [ systemd-journald ] <----------------------- [ /dev/kmsg ]
      |  (바이너리 데이터 생성)
      |
      | (D) /run/systemd/journal/syslog (Unix Domain Socket)
      |     : journald가 rsyslog에게 로그를 던져주는 전용 통로
      v
  [ rsyslogd ]
      |  (설정 파일: /etc/rsyslog.conf)
      +------> [ /var/log/messages ] (파일 저장)
      +------> [ TCP/UDP 514 포트 ]   (원격 전송)
```
### 커널 메시지
- 리눅스 커널에서 발생한 메시지가 /`dev/kmsg`라는 장치 파일을 통해 나옴  
  -> journald는 파일을 계속 감시하며 데이터를 가져옴

### 프로세스 로그
- 서비스가 실행될 때의 출력물을 systemd가 관리하고 이를 journald에 전달

### 유닉스 도메인 소켓 
- journald가 수집한 로그를 `/run/systemd/journal/sysslog`라는 소켓 파일로 다시 내보냄  
  -> rsyslog는 실행되자마자 소켓을 리스닝 하고 있다가 데이터가 들어오면 활용 (직렬화로 형태 변경)

### 로그 관리 분류

### `systemd-journald` : 휘발성/바이너리 로그
- 목적 : 현재 어떤 일이 발생했는지에 대한 확인 
- 구조 : key-value 형태의 데이터베이스
- 데이터 크기 : 메타데이터(PID, 인증 정보 등)가 많아 용량이 큼
- 변조 : 바이너리 형태라 임의 수정이 어려움 

### `rsyslog` : 영구적/텍스트 로그
- 목적 : 누군가 확인하거나 외부 서버로 보내기 위함
- 구조 : 시간순 정렬된 단순 텍스트 줄
- 데이터 크기 : 필요한 메시지 본문 위주로 작음
- 변조 : 일반 에디터로 수정 가능

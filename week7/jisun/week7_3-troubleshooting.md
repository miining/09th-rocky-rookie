# 서비스 장애 분석을 위한 로그 확인

## 장애 분석 시 로그 흐름 및 활용 구조
``` 
[ 서비스 장애 발생!! ]
               |
               v
+-----------------------------+
|    (1) rsyslog (텍스트)     | <--- "무슨 서비스가 죽었나?" (거시적 확인)
|  /var/log/messages 조회      |      (tail, grep 활용)
+-----------------------------+
               |
               v
+-----------------------------+
|    (2) journalctl (바이너리) | <--- "정확히 왜 죽었나?" (미시적 확인)
|  상세 메타데이터/종속성 분석   |      (journalctl -u 서비스명)
+-----------------------------+
               |
               v
+-----------------------------+
|    (3) 분석 결과 도출        | <--- OOM Kill, Segfault, 권한 문제 등
|   (조치 및 서비스 재시작)     |      원인 파악 완료
+-----------------------------+
```

## 장애 분석 명령어

### 최근 발생한 에러 확인
``` bash
$ journalctl -p err..emerg
```
- 시스템 전체에서 가장 심각한 에러만 필터링하여 우선순위 결정

### 특정 서비스 장애 추적
``` bash
$ journalctl -u [서비스명] -xe
```
- -xe : 설명과 함께 끝부분부터 출력 (에러 코드의 의미까지 설명)

### 부팅 실패 원인 분석 
``` bash
$ journalctl -b -1 -p err
```
- 이전 부팅 (-b -l)시 발생한 에러를 추적하여 부팅 불가 원인 파악

### 특정 시간 상황대 복기
``` bash
$ journalctl --since "10 min ago"
```
- 장애 발생 직전 10분간의 모든 시스템 변화(네트워크, 디스크 등) 확인

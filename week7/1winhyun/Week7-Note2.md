# 시스템 성능 분석 및 하드웨어 리소스 최적화 스터디 가이드

성능 분석을 할 때는 **USE 방법론(Utilization 사용률, Saturation 포화도, Errors 오류)** 을 기억하면 좋습니다. 자원을 얼마나 바쁘게 썼는지(사용률), 처리하지 못하고 대기열에 쌓였는지(포화도), 에러가 발생했는지를 중심으로 접근합니다.

---

## 1단계: 하드웨어 리소스 점검 도구 (Hardware Info Tools)

서버에 접속하여 가장 먼저 해야 할 일은 시스템이 어떤 하드웨어 스펙을 가지고 있는지 파악하는 것입니다.

### `lscpu`: CPU 아키텍처 및 토폴로지 확인

```bash
lscpu
```

CPU 아키텍처(x86_64, aarch64 등), 총 논리적 CPU 코어 수, 소켓 당 코어 수, 스레드 수, 가상화 지원 여부(VT-x/AMD-V)를 한눈에 보여줍니다. VM(가상머신) 내부인지 물리 서버인지도 하이퍼바이저 벤더 항목을 통해 확인할 수 있습니다.

### `lsblk`: 블록 디바이스(스토리지) 구조 확인

```bash
lsblk
lsblk -f    # 파일 시스템 타입(ext4, xfs 등)과 UUID까지 출력
```

시스템에 연결된 하드 드라이브, SSD, 파티션, LVM 구성, 마운트 지점을 트리 구조로 보여줍니다. `-f` 옵션은 스토리지를 영구 마운트할 때 유용합니다.

### `lspci`: PCI 버스 장치 확인

```bash
lspci
lspci -k    # 현재 사용 중인 커널 드라이버까지 출력
```

네트워크 카드(NIC), 스토리지 컨트롤러, 그래픽 카드 등 메인보드 PCI 슬롯에 장착된 장치를 나열합니다. `-k` 옵션은 드라이버 문제 진단에 핵심적입니다.

### `lsusb`: USB 장치 확인

```bash
lsusb
```

연결된 USB 버스와 장치(키보드, 마우스, 동글, 서버용 KVM 등)의 정보를 출력합니다.

### `dmidecode`: 메인보드 및 메모리 상세 정보 확인 (루트 권한 필요)

```bash
sudo dmidecode -t memory    # 메모리 슬롯별 상세 정보
sudo dmidecode -t bios      # BIOS 버전 및 펌웨어 정보
```

시스템 펌웨어(SMBIOS/DMI 테이블) 데이터를 읽어 하드웨어의 상세 정보를 출력합니다. 메모리 뱅크별로 어떤 제조사의 몇 GB RAM이 장착되었는지, 최대 지원 속도는 얼마인지 확인할 때 매우 유용합니다.

### `dmesg`: 커널 부팅 및 하드웨어 인식 로그 확인

```bash
dmesg -T --level=err,warn
```

부팅 과정이나 운영 중 커널이 하드웨어를 인식하면서 발생시킨 메시지(디스크 인식, 네트워크 링크 상태 등)를 보여줍니다. 장치 장애 시 하드웨어 에러 로그를 찾는 첫 번째 단계입니다.

## 2단계: CPU 및 프로세스 사용량 점검

CPU 사용률은 단순히 '몇 %를 쓰고 있나'가 아니라, '어떤 작업을 하느라 바쁜지'를 파악하는 것이 중요합니다.

### `top` / `htop`: 실시간 시스템 상태 및 프로세스 요약

```bash
top     # 종료할 때는 q 입력
htop    # 색상 적용된 트리 뷰 (EPEL 설치 필요)
```

시스템 요약 정보와 CPU/메모리를 가장 많이 소모하는 프로세스 목록을 실시간(기본 3초 간격)으로 보여줍니다. htop은 부모-자식 프로세스 관계를 트리 형태로 더 쉽게 볼 수 있습니다.

**핵심 CPU 지표 읽기 (top 상단)**

| 지표 | 설명 |
|---|---|
| `us` (User) | 사용자 프로세스(애플리케이션)가 CPU를 사용한 시간 |
| `sy` (System) | 커널(시스템 콜 등)이 CPU를 사용한 시간 |
| `id` (Idle) | 아무 작업도 하지 않고 쉬고 있는 잉여 시간 |
| `wa` (Wait) | 디스크 I/O 작업이 끝나기를 기다리는 시간. **이 값이 높으면 디스크 병목을 의심** |
| `st` (Steal) | 가상 환경에서 하이퍼바이저가 다른 VM을 처리하느라 빼앗아 간 시간 |

### `uptime`: 시스템 부하(Load Average) 확인

```bash
uptime
```

시스템 구동 시간과 함께 **Load Average(1분, 5분, 15분 평균)** 를 보여줍니다. 부하량은 '실행 중이거나 디스크 작업을 기다리는 프로세스의 개수'입니다. 이 수치가 논리적 CPU 코어 수(`nproc`)보다 지속적으로 높다면 CPU가 포화 상태임을 의미합니다.

### `pidstat`: 개별 프로세스/스레드별 CPU 사용률 상세 확인

```bash
pidstat 1 5     # 1초 간격으로 5번 측정
```

sysstat 패키지 도구로, 각 프로세스가 CPU 리소스를 어떻게 사용하고 있는지 상세히 분리하여 보여줍니다.

## 3단계: 메모리 상태 점검

리눅스는 빈 메모리를 적극적으로 디스크 캐시에 활용합니다. 따라서 메모리 지표는 올바르게 해석하는 법을 알아야 합니다.

### `free`: 메모리 사용량 간단히 보기

```bash
free -h     # GB, MB 단위로 읽기 쉽게 출력
```

**핵심 지표**

| 컬럼 | 설명 |
|---|---|
| `buff/cache` | 디스크 I/O 속도를 높이기 위해 임시로 사용 중인 캐시 메모리. 메모리 부족 시 자동 반환됨 |
| `available` | **가장 중요한 지표.** 애플리케이션이 실질적으로 추가 사용할 수 있는 여유 메모리(`free` + 반환 가능한 `buff/cache`). `free`가 적다고 메모리가 부족한 것이 아님 |

### `vmstat`: 가상 메모리 및 시스템 리소스 흐름 모니터링

```bash
vmstat 1 5
```

프로세스, 메모리, 스왑, I/O, CPU의 통계를 출력합니다.

**핵심 지표**

| 지표 | 설명 |
|---|---|
| `r` (runnable) | 실행을 위해 CPU를 기다리는 프로세스 수. 코어 수보다 크면 CPU 병목 |
| `b` (blocked) | 디스크 I/O 등을 기다리며 멈춰있는 프로세스 수 |
| `si`, `so` (Swap In/Out) | 물리적 RAM이 부족하여 디스크 스왑 영역과 데이터를 교환하는 양 |

> **⚠️ 수정 사항**: 원문에서는 `si`/`so` 값이 "지속적으로 0보다 크면 Thrashing"이라고 설명했으나, 이는 **과장된 표현**입니다.
>
> **Thrashing**은 시스템이 실제 유용한 작업보다 스와핑에 더 많은 시간을 소비하는 극단적인 포화 상태를 말합니다. `si`/`so` 값이 간헐적으로 발생하는 것은 메모리 압박의 신호이지만, Thrashing이라고 판단하려면 해당 수치가 지속적으로 높고 시스템 응답이 심각하게 저하되는 상황이어야 합니다. 어쨌든 `si`/`so` 값이 지속적으로 0보다 크다면 메모리 증설을 검토해야 합니다.

## 4단계: 디스크 I/O 성능 측정

서버 병목 현상의 가장 흔한 원인은 디스크 응답 지연입니다.

### `iostat`: 디스크 입출력 통계 분석

```bash
iostat -x 1     # 확장 정보(extended stats) 1초 단위 모니터링
```

블록 디바이스(디스크)의 성능을 분석하는 sysstat의 기본 도구입니다.

**핵심 지표 (RHEL 10 sysstat 12.x 기준 `iostat -x` 확장 모드)**

> **⚠️ 수정 사항**: 원문에서는 `kB_read/s`, `kB_wrtn/s`를 설명했으나, 이는 기본 모드(non-extended)의 컬럼명입니다. `-x` 확장 모드에서는 아래 표와 같이 컬럼명이 다릅니다.

| 지표 | 설명 |
|---|---|
| `r/s`, `w/s` | 초당 처리한 디스크 읽기/쓰기 요청 건수 |
| `rkB/s`, `wkB/s` | 초당 디스크 읽기/쓰기 대역폭(KB) |
| `r_await`, `w_await` | 읽기/쓰기 요청이 처리되기까지 대기한 평균 시간(밀리초). **이 값이 높으면 디스크 지연 병목** |
| `%util` | 디스크가 I/O 작업을 수행하느라 바빴던 시간의 비율. **100%에 근접하면 포화 상태** |

`r_await`/`w_await`이 높고 `%util`이 낮다면, 디스크 자체보다 스토리지 컨트롤러나 RAID 구성 문제를 의심해야 합니다.

### `biotop` (BCC 툴): I/O를 유발하는 주범 프로세스 찾기

```bash
sudo /usr/share/bcc/tools/biotop
```

`iostat`이 디스크 전체의 로드를 보여준다면, `biotop`은 eBPF 기술을 사용하여 **어떤 프로세스가 디스크 I/O를 가장 많이 발생시키는지** 실시간으로 순위를 매겨 보여줍니다. (패키지 `bcc-tools` 사전 설치 필요)

## 5단계: 종합 시스템 성능 이력 관리 및 고급 도구

문제가 발생한 시점에 항상 터미널을 보고 있을 수는 없습니다. 과거 데이터를 추적하고 커널 레벨의 프로파일링을 수행하는 도구입니다.

### `sar` (System Activity Reporter): 시스템 성능 과거 이력 조회

sysstat 패키지의 핵심으로, 백그라운드에서 주기적으로 지표를 수집하여 저장합니다. 과거의 특정 시간에 서버가 왜 느렸는지 분석할 때 필수적입니다.

```bash
sar -u          # CPU 사용량 이력 확인
sar -r          # 메모리 사용량 이력 확인
sar -d -p       # 디스크 I/O 이력 확인 (영구 장치명으로 표시)
sar -n DEV      # 네트워크 인터페이스 트래픽 확인
```

### `perf`: CPU 및 커널 정밀 프로파일링 도구

```bash
sudo perf top
```

시스템의 하드웨어 성능 카운터(PMU)를 이용하여 명령어 수준까지 성능 병목 현상을 추적합니다. 현재 시스템에서 CPU 자원을 가장 많이 소모하고 있는 **함수(Function)나 심볼** 단위의 목록을 실시간으로 보여줍니다. 애플리케이션의 특정 코드가 오버헤드를 일으키는지 디버깅할 때 사용합니다.

## 스터디 요약: 상황별 성능 분석 체크리스트

| 상황 | 확인 도구 |
|---|---|
| 처음 시스템 인계받을 때 | `lscpu`, `lsblk`, `dmidecode` 로 기준 사양 문서화 |
| 애플리케이션 구동 후 느려질 때 | `top` 으로 CPU, `free -h` 로 `available` 메모리 점검 |
| `top`에서 `wa`(I/O 대기)가 높을 때 | `iostat -x 1` 로 `%util` 100% 근접 디스크 파악 → `biotop` 으로 원인 프로세스 색출 |
| 밤새 서버가 느려졌다는 보고를 받았을 때 | `sar -u`, `sar -r` 로 과거 시간대 CPU/메모리 이력 분석 |
| 원인 불명의 성능 저하 | `sudo perf top` 으로 커널/함수 수준 병목 프로파일링 |
